这一步说明之前的 `libero` 路径已经修好了。现在缺的是 OpenVLA-OFT 模拟器侧的某个模块，脚本隐藏了具体模块名。图中的 TensorFlow、cuDNN、cuFFT、TensorRT 信息都是警告，不是本次退出原因。

先确认这个文件是否在文档预期位置：

```bash
cd "$MOWE_REPO_ROOT"

EXPECTED_OFT="$MOWE_REPO_ROOT/external/openvla-oft"
ls -l "$EXPECTED_OFT/experiments/robot/libero/libero_utils.py"
```

如果提示文件不存在，搜索实际位置：

```bash
find /home/ma-user/work/algorithm/chaoxintao_2/MoWE \
  -type f \
  -path '*/experiments/robot/libero/libero_utils.py' \
  -print
```

若找到了，例如：

```text
/home/ma-user/work/algorithm/chaoxintao_2/MoWE/openvla-oft/experiments/robot/libero/libero_utils.py
```

则目录少/多了一层。设置正确根目录：

```bash
export OPENVLA_OFT_ROOT=/home/ma-user/work/algorithm/chaoxintao_2/MoWE/openvla-oft

export PYTHONPATH="$OPENVLA_OFT_ROOT:$LIBERO_IMPORT_ROOT:$MOWE_REPO_ROOT:${PYTHONPATH:-}"
```

然后直接执行下面的真实导入，让 Python 显示被脚本隐藏的具体错误：

```bash
export MUJOCO_GL=osmesa
export PYOPENGL_PLATFORM=osmesa
export LIBGL_ALWAYS_SOFTWARE=1

cd "$MOWE_REPO_ROOT"

"$MOWE_PYTHON" - <<'PY'
import os
import sys
from pathlib import Path

oft = Path(os.environ["OPENVLA_OFT_ROOT"]).resolve()
assert (oft / "experiments/robot/libero/libero_utils.py").is_file(), oft
sys.path.insert(0, str(oft))

print("Python:", sys.executable)
print("OpenVLA-OFT:", oft)

from experiments.robot.libero.libero_utils import (
    get_libero_dummy_action,
    get_libero_env,
    get_libero_image,
    get_libero_wrist_image,
    quat2axisangle,
)

print("OpenVLA-OFT LIBERO simulator imports: OK")
PY
```

这里会出现两种结果：

- 显示 `OK`：就是 OpenVLA-OFT 路径少了一层，保持上述 `PYTHONPATH`，再运行 simulator smoke。
- 显示 `No module named 'xxx'`：`xxx` 才是真正缺少的依赖。需要将这个输出发给我再确定安装方式，不建议直接安装一批最新版依赖。

如果实际文件就在预期位置，可以这样设置：

```bash
export OPENVLA_OFT_ROOT="$MOWE_REPO_ROOT/external/openvla-oft"
export PYTHONPATH="$OPENVLA_OFT_ROOT:$LIBERO_IMPORT_ROOT:$MOWE_REPO_ROOT:${PYTHONPATH:-}"
```

然后重跑上面的真实导入检查。当前最可能的问题是 `external/openvla-oft` 的实际位置与 `stage3.json` 中的相对路径不一致，或者 `libero_utils.py` 内部还缺一个 Python 包。整个过程不需要 `sudo`，也与 EGL 无关。
